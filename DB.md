# DB Overview

## Обобщённое описание базы данных

В проекте используется реляционная база данных MySQL с двумя связанными таблицами:

- `firstmany` — основная таблица, в которой хранится главная сущность.
- `secondmany` — дочерняя таблица, в которой хранятся связанные записи.

Тип связи между таблицами: `one-to-many` (`1:N`).

Это означает:

- одна запись в `firstmany` может иметь много записей в `secondmany`;
- каждая запись в `secondmany` принадлежит только одной записи в `firstmany`;
- связь реализована через внешний ключ `secondmany.firstsolo_id -> firstmany.id`.

## Логика структуры

Таблица `firstmany` хранит базовую информацию о сущности:

- идентификатор;
- название;
- дату;
- текстовые поля;
- статус;
- числовые значения;
- JSON-данные;
- служебные поля времени создания и обновления.

Таблица `secondmany` хранит зависимые элементы:

- идентификатор;
- тип записи;
- ссылку на родительскую сущность;
- количество и приоритет;
- цену;
- логический флаг;
- дату и время;
- текстовые и бинарные данные;
- служебные поля времени создания и обновления.

## Ограничения и целостность

- `PRIMARY KEY` используется для уникального идентификатора каждой записи.
- `FOREIGN KEY` связывает дочернюю таблицу с основной.
- `ON DELETE CASCADE` удаляет дочерние записи автоматически при удалении родительской.
- `UNIQUE` используется для полей, значения которых не должны повторяться.
- `DEFAULT` задаёт значения по умолчанию.
- `INDEX` ускоряет поиск и фильтрацию по часто используемым полям.
